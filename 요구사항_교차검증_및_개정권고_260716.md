## 1. 검토 목적과 결론

본 문서는 기존 요구사항 정의서가 아이디어 제안 원문과 현재 프로토타입을 충분히 반영하는지 교차검증하고, 다음 세 가지 사용자 지적 사항을 정식 요구사항으로 보완하기 위해 작성한다.

1. 고객의 방문 일정 변경 기능
2. 개인 고객뿐 아니라 회사·사무실 등 기업 고객 관리
3. 방문기사가 설치·A/S·수리 작업을 완료하고 고객 확인 서명을 받는 기능

교차검증 결과는 다음과 같다.

- **방문 일정 변경은 직접 누락이면서 기존 제약과 충돌한다.** 제안 원문에는 고도화 기능으로 일정 변경과 방문예약 연계가 있지만, 요구사항 초안 `CR-005`는 방문예약 변경을 제외한다.
- **기업 고객 관리는 요구사항과 제안 원문 모두에 구조적으로 누락되어 있다.** 기존 `고객 → 제품` 구조만으로는 조직, 사업장, 현장 담당자와 복수 제품을 안전하게 표현할 수 없다.
- **설치·A/S·수리 구분과 서명은 직접 누락이다.** 기존 `FR-029`는 일반적인 방문 결과만 정의하며 작업지시, 작업 유형, 서명자와 서명된 완료 보고서를 정의하지 않는다.
- **제안서의 정량 AI 평가 목표와 운영 지표 일부가 정식 요구사항으로 이관되지 않았다.** 위험도 Recall, RAG Top-3, 안전 전환율, 인계 누락률, 케어 이행률과 반복 증상 지표를 보완해야 한다.
- 현재 프로토타입에는 지적된 세 기능의 주요 화면 흐름이 추가되었지만, 브라우저 로컬 저장 기반이므로 실제 인증·다중 사용자 동시성·법적 전자서명·영구 보존을 충족하는 운영 시스템은 아니다.

## 3. 직접 누락 및 충돌 사항

### 3-1. 방문 일정 변경

기존 요구사항은 상담사가 기사와 방문 예정일을 최초 등록하는 흐름만 정의한다. 고객이 날짜를 변경하거나 취소하는 기능, 관계자가 이를 승인·반려하는 기능, 변경 전후 일정 이력과 중복 요청 방지가 없다.

또한 `CR-005`가 실제 기사 배정뿐 아니라 방문예약 변경 자체를 제외하고 있어 사용자 요청과 직접 충돌한다. 외부 예약 API 연동 제외와 내부 시연용 일정 변경 제외를 분리해야 한다.

### 3-2. 기업 고객 관리

기업 고객은 개인 고객과 달리 다음 관계가 필요하다.

```text
서비스 계정
  └─ 조직
      ├─ 사업장 1..N
      ├─ 담당자 1..N
      └─ 구독·설치 제품 1..N
```

조직명만 고객 객체에 추가하는 것으로는 사업장별 방문 가능 시간, 출입 안내, 담당자 권한, 자산번호와 복수 제품을 관리할 수 없다. 문의와 방문은 최소한 `조직 ID`, `사업장 ID`, `제품 ID`, `담당자 ID`와 연결되어야 한다.

### 3-3. 현장 작업과 고객 서명

설치는 증상 문의 없이도 발생할 수 있으므로 모든 현장 작업을 문의 ID에만 종속시키면 안 된다. `작업지시 ID`를 독립적으로 관리하고 A/S·수리 건에서만 문의 ID를 선택적으로 연결하는 구조가 적합하다.

서명은 작업 결과에 대한 고객 확인 증빙이며 제품 상태가 완전히 해결되었다는 보증이나 권리 포기를 의미하지 않아야 한다. 서명 전 보고서 내용, 동의 문구 버전, 서명자와 시각을 보존하고, 서명 후 보고서가 변경되면 새 버전과 재서명이 필요하다.

### 4-1. 권장 상태 모델

문의 상태, 방문 실행 상태와 일정 변경 상태를 하나의 코드에 섞지 않는다.

```text
문의 상태: WAITING_COUNSEL → IN_COUNSEL → VISIT_SCHEDULED → VISIT_COMPLETE → COMPLETED
방문 상태: SCHEDULED → IN_PROGRESS → SIGNATURE_PENDING → COMPLETED
일정 요청: REQUESTED → APPROVED | REJECTED | CANCELLED
```

승인 시에만 `visit.scheduledAt`을 변경하고 일정 버전을 증가시킨다. 완료 방문, 과거 희망 시간, 현재 확정 시간과 동일한 요청, 중복 대기 요청은 차단한다.

## 11. 운영 전환 경계

현재 결과물은 **사업형 UI와 역할 간 업무 흐름을 검증하는 정적 프로토타입**이다. 다음 항목을 완료하기 전에는 실제 고객·기업·기사 업무에 사용할 수 없다.

### 11-1. 프로토타입에서 검증 가능한 범위

- 개인·기업 고객 구분과 기업 사업장 정보 표시
- 고객이 복수 제품 중 문의 제품을 선택하는 흐름
- 고객 일정 변경 요청과 관계자 승인·반려 흐름
- 설치·A/S·수리·정기 케어 작업 유형 선택
- 방문기사 작업 결과와 고객 서명 입력
- 고객·상담사·기사 화면 간 동일 브라우저 상태 연결
- 합성 데이터 기반 변경 이력과 케어 이력 반영

### 11-2. 운영 전환 전 필수 항목

1. 서버 인증·인가와 고객/조직/사업장 단위 접근통제
2. 관계형 DB와 서버 트랜잭션, 일정 버전과 낙관적 잠금
3. 실제 기사 가용시간·예약·알림 시스템 연동 정책
4. 개인정보·서명정보 수집 동의, 보존기간, 암호화와 파기 정책
5. 서명된 작업 보고서의 버전·해시·재서명과 법무 검토
6. 실제 업무에 필요한 작업지시, 부품·재고, 보증·견적 시스템의 연동 또는 명시적 제외
7. 감사 로그의 변경 불가 보관과 운영 모니터링
8. 백업·복구, RPO·RTO와 장애 대응 훈련
9. 웹 접근성, 모바일 실기기, 브라우저 호환성과 보안 시험
10. AI 품질·안전 평가 세트와 정량 수용 기준 통과

### 11-3. 현재 구조의 명시적 한계

- `localStorage`와 `BroadcastChannel`은 같은 브라우저의 탭 간 시연용이며 팀원의 서로 다른 PC를 공유 DB처럼 연결하지 않는다.
- 화면의 역할 전환은 인증이나 서버 인가가 아니다.
- 서명 점 데이터와 `integrityId`는 법적 전자서명, 공인 시각확인 또는 위변조 방지 해시를 대신하지 않는다.
- 브라우저 데이터 초기화 시 기록이 사라질 수 있으므로 영구 보존과 백업 요구를 충족하지 않는다.
- 가상 기업·고객·제품·일정·문서 메타데이터를 실제 업무 데이터로 사용해서는 안 된다.

## 12. 최종 권고

이번 개정에서는 `FR-038~045`, `NFR-019~027`, `DR-016~022`와 `CR-005` 수정, `CR-013~015` 추가를 요구사항 베이스라인 후보로 채택하는 것이 적절하다.

다만 프로토타입 완성도와 운영 준비도를 분리해서 판단해야 한다.

- **프로토타입 완료 판단:** 사용자 지적 세 기능의 역할별 화면과 상태 흐름이 합성 데이터로 연결되고 검증됨
- **운영 준비 판단:** 인증·권한·서버 DB·동시성·개인정보·서명 무결성·백업·정량 AI 평가가 모두 충족됨

따라서 현재 결과물은 “실제 사업에서 사용할 법한 업무형 프로토타입”으로는 적합하지만, “실제 고객 데이터를 처리하는 운영 서비스”로 표현해서는 안 된다.
